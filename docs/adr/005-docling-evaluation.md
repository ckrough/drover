# ADR-005: Docling Evaluation Spike — Reject as Drop-In Loader Replacement

## Status

Accepted (2026-04-29). Closes the spike scoped in `docs/research/docling-evaluation-spike.md`. Drover keeps `unstructured` as the default loader. The optional `[docling]` extra and the `DROVER_LOADER=docling` switch land in tree as opt-in research surface; they are not enabled by default and are not on the supported configuration list.

**Amended (2026-04-30):** the spike plumbing this ADR's "What stays" section described as "kept in tree as research surface" has been fully removed on branch `nostalgic-engelbart`. The decision (reject) is unchanged — see the empirical results below, which justified rejection on five of seven gating triggers (LLM category lift, LLM doctype regression, format parity, per-doc loader latency, and partial network-call failure). Specifically removed: the `[docling]` install extra, `DoclingLoader`, `LoaderType` enum, `--loader` CLI flag, `--debug-structure` flag, `DROVER_LOADER` env var, `LoadedDocument.docling_doc`, `loader_latency_ms` / `loader_backend` instrumentation, the format-coverage matrix harness (`scripts/format_matrix.py`, `tests/test_format_matrix.py`, `eval/format_matrix.md`), the benchmark runner (`scripts/benchmark_docling_loaders.py`, `eval/runs/`), and the OCR integration test. The re-evaluation criteria in this ADR still apply; the bar is now higher because the harness must be rebuilt before re-measurement. See the cleanup commits on `nostalgic-engelbart` for the deletion (refs beads `prof-wis`).

## Context

ADR-002 commits drover to a privacy-first, local-first architecture; ADR-004 then standardized on a local LLM (Ollama gemma4:latest) as the primary classification path. With the LLM saturating most axes on the 80-document eval corpus (87.5% domain / 86.2% doctype on flat text), the remaining accuracy ceiling sits on category accuracy at 40.0%. The hypothesis driving this spike was that the loader, not the model, is the bottleneck on category — that flattening structured documents to text discards the section / table cues that disambiguate `tax / property` from `tax / income`, and that a structure-aware loader would lift category toward parity with domain and doctype.

Docling (IBM Research, MIT, LF AI & Data Foundation) is the current best-in-class structure-aware loader. It produces a `DoclingDocument` with hierarchical headings, parsed tables, reading order, and a `HybridChunker` that respects structure. It also extends format coverage (audio via WAV/MP3, USPTO/JATS XML, LaTeX) and ships an OCR fallback for scanned PDFs.

The spike instrumented all three downstream consumers (LLM prompt, NLI chunker, metadata extractors) and ran the full 80-doc corpus under both loaders for three providers, then assessed go/no-go against the spec's success metrics.

| Loader / chunker | Domain | Category | Doctype | Vendor | Date |
|---|---|---|---|---|---|
| unstructured / sliding+mean | 2.5% | 2.5% | 6.2% | 1.3% | 77.5% |

Docling's section-aware chunks lift NLI on most axes by single digits. The gap to the LLM path remains 80+pp on every classification axis, unchanged in operational terms (consistent with the spec's prediction that "Docling helps NLI more than it helps LLM in relative terms, but not enough to flip the operational recommendation").

### Format coverage matrix

The 7 GAIN rows are real — Docling's RapidOCR pipeline extracts from image-only files where unstructured (without poppler / OCR extras) returns nothing. The 3 REGRESS rows are blocking input under the spec.

### Runtime cost

- **Average per-document loader latency**: ~57 ms (unstructured) → ~2,400 ms (docling). Roughly 42x for gemma4 / 32x for Haiku.
- **First-run model downloads**: Docling's RapidOCR auto-fetches three ONNX models (~15 MB total) from `modelscope.cn` on first invocation. Subsequent runs are offline. This is a one-time cost that nonetheless violates the spec's "no runtime network call" expectation under ADR-002 unless we ship the models pre-downloaded.
- **Total wallclock**: gemma4 + docling = 91 min vs gemma4 + unstructured = 91 min on the same Ollama instance (the LLM call dominates and queues serially); haiku + docling = 4 min 48 s vs haiku + unstructured = 1 min 54 s (loader latency is visible because the API call is fast).

### Dependency footprint

The `[docling]` extra introduces (per `uv sync --extra docling` 2026-04-29):

- `docling`, `docling-core`, `docling-parse`, `docling-ibm-models` (the core stack).
- ONNX runtime (`onnxruntime`) for layout / OCR inference.
- `rapidocr` (MIT, ONNX) — bundled OCR engine.
- `tree-sitter` and language grammars (Python / JavaScript / TypeScript / C) — used for code-aware parsing.
- `easyocr` and `pillow-heif` are opt-in transitives, not pulled by default.

License audit: all transitives observed are MIT / Apache-2.0. No GPL / AGPL surprises.

## Evaluation Against Spec No-Go Triggers

| Trigger | Threshold | Observed | Result |
|---|---|---|---|
| LLM category lift | ≥ +5pp | gemma4: −3.8pp; haiku: −1.3pp | **FAIL** |
| LLM doctype regression | ≥ −2pp | gemma4: −2.5pp | **FAIL** |
| LLM domain regression | ≥ −2pp | gemma4: 0.0pp; haiku: +1.3pp | pass |
| Format parity | 0 REGRESS rows | 3 REGRESS (`.eml`, `.epub`, `.rtf`) | **FAIL** |
| Per-doc loader latency | < 5x current | ~42x | **FAIL** |
| MIT-only deps | required | satisfied | pass |
| No runtime network calls | required | first-run RapidOCR model fetch | partial — bundleable with offline cache |

Five of seven triggers fail. The headline (category lift) is the one the spike was built to test, and it failed in both directions: gemma4 regressed by 3.8pp and Haiku — which already extracts category at 55% on flat text — was unmoved by Docling's markdown.

### Why the structural-input hypothesis didn't hold here

The most defensible reading: the 80-doc corpus is synthetic, generated via Claude Sonnet 4.6 with structured markdown templates per doctype (real headings, pipe tables, signature blocks, line items per `scripts/generate_eval_samples.py`). When that markdown is rendered to PDF and re-parsed by `unstructured`, the section text already lands flat in reading order with explicit heading text inline (e.g., "Bill To" on its own line, "INVOICE" as the first non-blank line). The LLM does not need a markdown re-export to see structural cues that the original rendering preserved. Docling's markdown export, meanwhile, restructures the same content into a different surface form — different enough to perturb the LLM's category-axis cues (dropping 3.8pp), without adding structural information the model could not already infer.

A real-world corpus of scanned, multi-column, multi-section documents (lease agreements, mortgage closing packages, scanned medical records) might invert this finding. The spike does not have such a corpus and adopting Docling on this evidence would be a straight optimism bet.

## Decision

**Decision: reject.**

Drover does not adopt Docling as the default document loader. `unstructured` remains the default. The optional `[docling]` extra and the `DROVER_LOADER=docling` switch (added in `prof-c26`) stay in tree as research surface for future work; they are not on the supported configuration list and are not exercised in CI by default.

### What stays (subsequently removed — see Status amendment)

The spike's plumbing has independent value and is kept:

- The `LoaderType` enum, `--loader` CLI flag, and `DROVER_LOADER` env var. Trivial cost; lets users opt in if they want to experiment.
- `DoclingLoader` and the lazy import path. No runtime cost unless the user passes `--loader docling`.
- `LoadedDocument.docling_doc`, the `structured_regions` extractor protocol extension, and `HybridChunker` integration. All gated on `docling_doc is not None`; no behavior change when it is `None`.
- `loader_latency_ms` and `loader_backend` instrumentation (`prof-jt0`). Useful for any future loader benchmark, not just Docling.
- The `--debug-structure` CLI flag (`prof-rbz`). No-op without `--loader docling`; useful for ad-hoc structure inspection.
- The format coverage matrix (`prof-dt6`) — `tests/test_format_matrix.py` and `scripts/format_matrix.py`. Reusable harness.
- The benchmark runner (`prof-nft`) — `scripts/benchmark_docling_loaders.py` and `eval/runs/`. Reusable harness; rerunning on a different corpus is one shell loop.

### What does not change

- Default `DROVER_LOADER` stays `unstructured`. No deprecation of `unstructured` is planned.
- ADR-004 stands: Ollama gemma4:latest remains the primary local classification path. The spike's gemma4 + docling result reinforces that — a stronger structural input did not unlock further accuracy on this corpus.
- Privacy-first defaults (ADR-002) are unchanged.

### What would change the decision (re-evaluation criteria)

A future re-run of this benchmark **on a non-synthetic corpus** that meets all of:

1. **At least 50 real-world documents** sourced outside `scripts/generate_eval_samples.py` (e.g., user-donated household paperwork or a public corpus of scanned forms).
2. **At least 30% of the corpus is structurally complex** in a way the synthetic generator does not produce: scanned multi-column forms, mortgage closing packages, lease agreements with embedded tables of fees, medical records with order tables.
3. **Either a +5pp category lift or a +5pp vendor lift** for gemma4 under Docling on that corpus.

If any of those is satisfied, this ADR should be reopened and a follow-up issue filed against the parent epic. Until then, Docling is held in tree but inactive.

### Selective adoption (out of scope, but considered)

A "Docling for image-only inputs, unstructured for everything else" hybrid loader was considered and rejected for this spike's scope. The 7 GAIN rows on image extensions are real, but drover's eval corpus has zero image-only inputs (all 80 docs are born-digital PDFs), so we cannot quantify the user-facing benefit. Selective adoption can be revisited if a real corpus surfaces image-only inputs that users care about classifying.

### Followups (subsequently removed — see Status amendment)

- `prof-rbz` debug structure flag — kept; uses can opt in to inspect a `DoclingDocument` JSON dump on demand.
- `prof-kjg` OCR scanned-PDF integration test — kept; opt-in via `DROVER_RUN_OCR_TEST=1`. Documents the OCR capability if a future ADR revisits selective adoption.
- `eval/baselines.md` and `eval/format_matrix.md` are now historical artifacts under the 2026-04-29 entry. They are not maintained against future Docling releases.

## Consequences

### Positive

- **No new default-path complexity.** Users continue to get the same loader they already had. No prompt re-tuning, no NLI hypothesis-template churn, no fallback / hybrid loader logic.
- **No 32-42x loader latency tax** on the default path. Drover keeps its ~57 ms / doc loader budget.
- **No new runtime network requirement** (RapidOCR's first-run model fetch only triggers under `--loader docling`).
- **Format coverage stays at the existing baseline.** The 3 REGRESS rows (`.eml`, `.epub`, `.rtf`) remain handled by `unstructured`. Users do not lose support for these formats.
- **Re-evaluation is cheap.** ~~All harnesses (benchmark runner, format matrix, OCR test) stay in tree. A future re-run on a new corpus is `uv sync --extra docling && uv run python scripts/benchmark_docling_loaders.py ...`.~~ (Subsequently removed — see Status amendment. The harness must be rebuilt before re-measurement.)

### Negative

- **`[docling]` extra and its dependencies are dead weight in tree.** They install only when a user passes `--extra docling`. Default installs are unaffected, but the optional surface needs documentation and gets minimal CI coverage.
- **Image-only documents remain badly supported.** The 7 GAIN rows expose a real `unstructured` weakness. We accept this for now because the corpus does not exercise it; user reports may revisit it.
- **No category-axis ceiling break.** Category at 40% on gemma4 stays the LLM's ceiling on this corpus. The spike eliminated "loader" as the bottleneck; the residual gap is now firmly attributable to model capacity (Haiku gets to 55%) and / or per-category prompt engineering (out of scope per ADR-001).

### Neutral

- **Selective hybrid (image-only via Docling) is filed as a future option, not a current commitment.** The harness would let us measure it; we have no real-image data to measure against today.
- **NLI's small Docling lift (+1-2.5pp on most axes) is recorded but does not change ADR-004.** The gap to the LLM path remains too large for NLI to be a competitive default.
